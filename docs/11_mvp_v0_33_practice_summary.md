# MeetingSummaryApp MVP v0.33 实践总结

## 版本定位

MVP v0.33 不是一个“继续堆功能”的版本，而是一次明确的稳定性收口版本。

如果说：

1. `v0.2` 解决的是“多人逐字稿和纪要主链能不能跑”
2. `v0.32` 解决的是“这个产品能不能持续被用”

那么：

`v0.33` 解决的是“在真实长会和多次连续使用里，这个产品会不会突然掉链子”。

这一版的核心不在模型效果本身，而在三条更偏产品稳定性的主线：

1. `录音主链容错`
   - 系统音频中断不再拖死整场录音
   - 启动失败、停止失败、重启恢复的状态语义收紧
2. `采集策略升级`
   - 从固定 display 捕获，升级为“窗口优先、自动重绑、失败降级”
3. `历史与结果页体验收口`
   - 历史筛选更精确
   - 标题、时间、空态、复制、错误展示更符合真实使用预期

因此，v0.33 的价值更接近“把产品从能用，推进到敢放心录长会”的一版。

## 本版本达成的目标

### 1. 系统音频中断不再等于整场录音失败

这是 v0.33 最重要的结论。

在真实使用里，之前最致命的问题不是模型偶尔失败，而是：

1. 录音中途系统音频流被系统停掉
2. 用户侧还以为在正常录音
3. 点停止时又触发一次 stop
4. 最终整场录音报废
5. 下一次录音还可能因为残留状态起不来

v0.33 把这个问题重新定义了：

1. `系统音频中断`
   - 只代表一条音源失效
   - 不代表整场录音结束
2. `麦克风仍在录`
   - 主状态就仍然是 `录音中`
3. `停止录音`
   - 仍然可正常执行
   - 最终至少能产出“仅麦克风版本”的结果

这是一个很关键的状态语义修正，因为它把“部分降级”从“整场失败”里剥离了出来。

### 2. 录音采集策略从“绑第一块屏”升级为“窗口优先 + 自动恢复”

之前系统音频录制路径的根问题是：

1. 默认拿 `SCShareableContent.current.displays.first`
2. 用 display filter 建流
3. 一旦屏幕参数变化，`SCStream` 很容易被系统结束

v0.33 做的不是简单补日志，而是把采集策略本身往前推了一步：

1. 启动录音时优先寻找会议主窗口
2. 如果找到候选会议窗口，则优先绑定 `desktopIndependentWindow`
3. 如果找不到稳定窗口，再回退到 display
4. 录音过程中持续监听共享内容变化
5. 原窗口消失但同一 App 出现新候选窗口时，自动尝试重绑
6. 重绑失败时再降级到 display 或仅麦克风

这意味着产品第一次具备了“采集目标可迁移”的能力，而不再是一次绑定、断了就死。

### 3. 启动失败、重试处理、失败历史的闭环补全了

这一版也补上了很多真实使用里很伤体验的小坑：

1. 启动失败会写入失败历史记录
   - 不再出现“明明失败过一次，但历史里根本没有这条”的情况
2. 引擎提供强制复位能力
   - 避免异常后卡在“已经处于录音状态”
3. 最近一次失败会话可以手动续跑
   - 即使系统音频损坏，也能回退成仅麦克风轨继续处理
4. 重跑后旧错误文案会被清掉
   - 不再出现“流程已经重新开始，但页面还挂着上一次 Gemma 报错”的情况

这些问题单看都不大，但它们加起来决定了产品在高频使用下是不是可信。

## 本版本新增/强化的能力

### 录音稳定性

#### 1. 系统音频中断后的状态机收口

现在录音状态被拆成了两层：

1. 整体录音状态
   - `starting / recording / paused / stopping / failed`
2. 分音源状态
   - 麦克风
   - 系统音频

这样 UI 才能准确表达：

1. 主状态仍是 `录音中`
2. 次状态显示 `系统音频已中断`
3. 并提示“当前仅继续录制麦克风”

这比过去直接掉成“录音失败”或“未录制”要合理得多。

#### 2. pause / resume / stop 的幂等性更强了

这一版修掉了几个很典型的运行时问题：

1. pause 时系统音频状态误标成 `idle`
2. 系统流已被系统停止时，再 stop 会被当成致命错误
3. 某些异常路径下内部状态没清理干净，下一次 start 会直接报 `alreadyRecording`

修复后，暂停、恢复、停止和异常路径的状态都更可预测，减少了“UI 看起来停了，底层却没停干净”的情况。

#### 3. 系统流诊断信息更完整

v0.33 这版对中断诊断的价值也很高。

当前日志里已经会记录：

1. `SCStream` 的 `domain`
2. `code`
3. 枚举语义
4. `userInfo`
5. 当前 display / window 绑定信息
6. 屏幕参数变化、睡眠/唤醒、会话状态变化

这使得“为什么突然中断”第一次从猜测，变成了可定位的系统级事实。

### 历史与结果页体验

#### 1. 历史筛选更像真正可用的桌面列表

v0.33 对历史区域的处理不只是美化，而是提升检索效率：

1. 时间筛选仍保留快捷项
2. 新增“精确日期”筛选
3. 状态筛选、搜索、时间筛选可以组合使用

这让会议数量积累起来后，用户能更快定位目标记录。

#### 2. 已完成会议标题开始从内容里提炼主题

之前历史标题长期是：

1. `4月13日 15:33 会议`
2. `4月13日 13:38 会议`

这在会议多了之后几乎不可用。

v0.33 里，已完成会议会优先从纪要内容里找真正的“会议主题”字段来展示，而不是长期保留默认时间标题。

这一步虽然还不是完整的“会议自动命名系统”，但已经明显提升了历史列表的可读性。

#### 3. 历史时间改成展示录制时间

这是一个很典型的产品语义修正。

用户在历史列表里更关心的是：

1. 这场会是什么时候开的

而不是：

1. 最后一次处理完成是什么时候

所以这一版把左侧列表展示的时间统一收敛成 `createdAt`，也就是会议时间/录制时间，而不是更新时间。

#### 4. 空态、复制、布局细节更一致

结果页这版做了几类收口：

1. 多人逐字稿空态和会议纪要空态统一
2. 会议纪要增加复制按钮
3. 复制按钮位置调整到内容区域右上角
4. 纪要内容顶部对齐和左侧逐字稿区域重新拉齐
5. 默认文案、边距和空白关系整体收紧

这些都是“看起来像细节”，但它们直接影响结果页像不像一个可以长期工作的界面，而不是临时 demo。

## 本版本最有价值的工程取舍

### 1. 没有继续把“系统音频失败”当作“整场失败”

这是 v0.33 最正确的取舍。

如果继续沿用之前的语义：

1. 用户会因为一条音源失败丢掉整场录音
2. 历史与处理链路都会被放大成失败
3. 真实使用里产品会显得非常脆

改成“部分降级、主链继续”之后，系统才更像一个真正能工作的录音产品。

### 2. 没有继续把采集目标写死在 `displays.first`

这也是正确取舍。

继续绑第一块屏虽然实现最简单，但在真实场景里风险太高：

1. 分辨率变化
2. 显示器切换
3. 会议窗口切小浮层
4. 屏幕参数变化

都会让系统流变脆。

改成“窗口优先、display 兜底”后，虽然还不是绝对稳，但方向已经从“硬编码 demo”走向“真实会议场景优化”。

### 3. 没有为了推 GitHub 而保留旧仓库的兼容壳

这也是一次工程上很重要的收口。

在把 GitHub 真源切回 `meetingSummary-main` 的过程中，中间出现过一层为旧目录结构补的兼容代码。  
最后明确把这批临时兼容层还原掉，只保留当前主工作区的真实结构，这是对的。

否则文档、代码、GitHub 三者会继续分叉，后面版本会越来越难维护。

## 本版本踩到的主要坑

### 1. `ScreenCaptureKit` 的系统音频并不是真正独立于环境的采集

这是这一版最大的现实教育。

一开始直觉上会以为：

1. 我只录系统音频，不录屏幕画面
2. 那系统音频应该和 display 没关系

但真实情况是：

1. `SCStream` 仍然要基于 `SCContentFilter`
2. filter 本质仍然依附 display 或 window
3. 因此屏幕参数变化依然可能让流被系统结束

这件事最终逼着这一版把设计从“绑显示器”推进到“目标窗口优先 + 自动恢复”。

### 2. 续跑流程和 UI 错误态不是一回事

最近一次失败会话在手动续跑时，后端实际已经重新入队并开始跑了，但页面上的旧错误文案没有清掉。

这暴露出一个很典型的问题：

1. 处理流程重启了
2. 但 `errorMessage` 这种展示态没同步清理

所以用户看到的是：

1. 后台在干活
2. 前台还挂着旧失败提示

这类问题说明：  
重试处理不仅是 pipeline 行为，也必须同步清理 UI 的错误语义。

### 3. Git 真源切换时，最危险的不是代码，而是结构分叉

这次把 GitHub 真源切回 `meetingSummary-main` 的过程中，也踩到了一个很典型的工程坑：

1. 当前工作目录不是 Git 仓库
2. 另一个目录连着 GitHub，但结构已经分叉
3. 如果粗暴整体覆盖，很容易把旧的脏改动、额外模块和兼容逻辑一起推上去

最后实际证明，最稳的路径是：

1. 先完成功能同步
2. 再把 Git 历史接回当前工作区
3. 最后把临时兼容层清掉，让真源重新单一

这比“先推上去再说”要稳得多。

## 当前版本边界

v0.33 虽然把稳定性往前推了一大步，但仍有明确边界：

1. `ScreenCaptureKit` 路线下，系统音频仍不可能做到绝对独立于桌面环境
2. 会议窗口识别目前仍是启发式，不是强约束绑定
3. 小浮层 / 画中画 / 浏览器会议页的窗口切换，仍可能存在误判或重绑失败
4. “仅麦克风保底”是降级策略，不是根治系统音频问题的最终方案
5. 历史标题提炼已经可用，但还不是严格的结构化主题抽取系统

换句话说，v0.33 解决的是“突然中断时别把整场拖死”，但还没有把“系统音频永不中断”彻底解决。

## 对下一阶段的建议

### 1. 继续把会议窗口识别做成正式策略

下一步最值得继续投入的是：

1. 更稳的窗口候选评分
2. 明确的会议 App 识别规则
3. 更细的自动重绑日志
4. 对浮层 / 画中画 / 浏览器会议的专项验证

### 2. 补一次完整的稳定性回归测试

建议至少系统性跑这几类场景：

1. 长会连续录制
2. 连续两次开始/停止
3. pause / resume
4. 切小浮层
5. 切换显示设置
6. 失败后重试处理

这一版的改动很多都在运行时状态机里，必须靠场景验证把质量锁住。

### 3. 继续收口“结果页像产品而不是工程面板”

目前结果页已经明显比之前好，但还可以继续推进：

1. 历史标题提炼进一步结构化
2. 纪要 Markdown 渲染细节继续增强
3. 复制、空态、版本切换提示进一步统一

## 当前结论

MVP v0.33 的价值，不是“再加一批功能点”，而是把产品里最容易在真实使用里翻车的几件事，真正收口了一轮。

本版本最重要的成果有五个：

1. 系统音频中断不再拖死整场录音
2. 录音状态机第一次真正支持“部分降级、主链继续”
3. 采集策略从固定 display 升级到窗口优先和自动恢复
4. 历史筛选、标题、时间语义和结果页交互更符合真实使用
5. GitHub 真源重新统一到了 `meetingSummary-main`

如果说 v0.32 解决的是“这个产品能不能持续被用”，  
那么 v0.33 解决的是“这个产品在高频和长会场景下，会不会突然失去可信度”。
