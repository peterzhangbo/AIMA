# MeetingSummaryApp MVP v0.32 实践总结

## 版本定位

MVP v0.32 不是一次单点修补，而是在 v0.2 的“多人逐字稿 + 会议纪要”基础上，继续把产品往“可持续使用的桌面会议系统”推进的一版。

这个版本的核心价值不在单个算法点，而在三条主线同时往前走：

1. `V3.1 系统能力`
   任务队列、历史记录、版本化、失败恢复
2. `V3.2 体验增强`
   浮层录音、主窗口切换、逐字稿编辑、播放联动
3. `长会议可用性`
   分段 ASR、rerun 机制、低信息文本过滤、纪要重生成与恢复点

因此，v0.32 更接近“第一版可长期使用的产品骨架”，而不是单次 demo。

## 本版本达成的目标

### 1. 录音与处理彻底解耦

在更早的 MVP 阶段，录音、转写、纪要生成更像是单次同步流程。  
v0.32 已经完成了向“会议任务系统”的过渡：

1. 每场会议都有独立 `MeetingRecord`
2. 录音结束后进入处理队列
3. 上一场处理时可继续录下一场
4. 历史记录从 SQLite 读取，不依赖当前 UI 状态

这意味着产品已经不再只是“录一次、看一次”的工具，而是开始具备持续积累会议资产的能力。

### 2. transcript / summary 版本化落地

v0.32 已支持：

1. 多人逐字稿版本
   - clean 基线版本
   - edited 人工修订版本
   - rerun / rerun2 等离线重跑版本
2. 会议纪要版本
   - 初始生成版本
   - 重新生成后的新版本

同时补上了这些配套能力：

1. 设为当前版本
2. 历史详情页切换版本
3. 最新版本默认展示
4. rerun 结果优先识别

这让“原始结果、优化结果、人工修订结果”之间的关系第一次真正可追踪。

### 3. 长会议开始可用

这一版最关键的实践收获，是长会议主链终于从“能跑”升级到了“基本能用”。

做了两件实事：

1. `长会议分段 ASR`
   - 超过阈值的长音频自动按 chunk 跑 Whisper
   - chunk 带 overlap
   - 最后回拼成统一 `transcript_raw.json`

2. `低信息文本过滤`
   - 过滤长段 filler
   - 压缩重复口癖
   - 裁掉首尾 filler

在真实 1 小时会议上，分段 ASR 明显提升了召回：

1. transcript segment 数量显著上升
2. 原来 diarization 里“明显有人在说话但没出字”的时间窗显著下降
3. rerun 结果在多人逐字稿和纪要层面都明显优于旧版

这说明方向已经被验证：  
长会问题不能只靠 Whisper 参数抖一抖，必须做分段策略。

## 本版本新增/强化的能力

### V3.1 系统能力

#### 1. 历史记录中心

左侧历史记录现在已经具备：

1. 搜索
2. 状态筛选
3. 时间筛选
4. 当前选中态高亮
5. 会议详情跳转

并且“开始新会议”入口已经固定在历史区底部，更符合桌面产品长期使用的操作模型。

#### 2. 队列与恢复

v0.32 已支持：

1. 会议入队
2. 重开 App 后恢复 `queued / processing` 队列
3. 失败会议重试
4. 更智能的“从失败点继续”

其中最重要的是：

`重试处理` 不再一律从头跑，而是基于：

1. `session.log`
2. 已归档产物文件

双重判断恢复点。

例如：

1. Whisper 已成功 -> 跳过 Whisper
2. diarization 已成功 -> 跳过 diarization
3. 多人逐字稿已成功 -> 跳过 merge / clean
4. 只剩 Gemma 失败 -> 直接从纪要生成恢复

这是本版本非常重要的工程质量提升，因为它显著减少了长会场景下的重复等待成本。

#### 3. 历史记录与归档文件脱钩

历史记录现在来自 SQLite，而不是实时扫描 session 文件夹。  
因此：

1. 删除 `~/Documents/MeetingSummaryApp/sessions/...` 某个会议目录
2. 历史索引不会自动消失

为了避免“索引还在但内容空掉”的坏体验，本版本新增了：

1. 归档缺失检测
2. 明确提示“归档文件已丢失”
3. “清除此记录”按钮

这样历史数据和归档文件的边界更清晰了。

### V3.2 体验增强

#### 1. 悬浮录音浮层

当前已实现：

1. 主窗口和浮层互斥
2. 开始录音后自动缩成浮层
3. 浮层可暂停 / 继续 / 结束 / 展开

这让产品开始符合真实使用场景：  
录音时不必一直把主窗口占在前台。

#### 2. 逐字稿点击回放与联动高亮

结果页已支持：

1. 点击逐字稿段落跳转回放
2. 当前播放位置联动高亮逐字稿
3. 进度条 seek

这让逐字稿从“静态结果”变成“可验证、可追溯的工作界面”。

#### 3. 逐字稿内联编辑

多人逐字稿区域已支持：

1. 修改发言人
2. 修改发言内容
3. 合并到上一段
4. 拆分段落

并且采用的是：

1. 就地进入编辑态
2. 失焦自动提交
3. 也可显式点击确认

这版设计比弹独立输入框更顺手，也更符合“逐字稿是工作材料”的使用方式。

#### 4. 版本切换体验

本版本对版本展示做了比较大的收敛：

1. transcript 不展示 raw
2. 第一版 clean 直接叫 `v1`
3. edited 顺延成 `v2 edited / v3 edited`
4. 版本切换改成下拉
5. 下拉默认展示最新版本
6. 当前版本用系统 checkmark 表示

这比之前把中间态暴露给用户要清晰得多。

## 本版本最有价值的工程取舍

### 1. 没有把“重试”和“重新生成纪要”混成一回事

这一点是对的。

现在两者语义已经明确分开：

1. `重试处理`
   - 用于失败会议
   - 从恢复点继续主链
2. `新建纪要`
   - 用于已完成会议
   - 基于当前 transcript 再生成一个新的 summary version

这能避免用户混淆，也让版本行为更干净。

### 2. 没有把 session 文件夹直接当数据库

这是对的。  
如果历史完全依赖目录扫描，会导致：

1. UI 状态不可控
2. 删除文件即删历史
3. 版本关系难管理

SQLite 存索引 + 文件系统存产物，这个分层是必要的。

### 3. 没有继续依赖“整份长会一次性喂模型”

这也是关键取舍。  
无论是 Whisper 还是 Gemma，一次性喂很长上下文都很脆：

1. Whisper 会漏召回
2. Gemma 会 OOM

因此：

1. Whisper 走分段 ASR
2. Gemma 输出上限重新收回保守值

这是理性方案，而不是盲目继续堆 token。

## 本版本踩到的主要坑

### 1. 长会纪要阶段 OOM

在一次长会议测试中，把 Gemma `max_tokens` 提到 `12000` 后，出现了：

1. prompt 很长
2. prefill token 很大
3. Metal / unified memory 不足

最终表现为：

`[METAL] Command buffer execution failed: Insufficient Memory`

这个问题最终说明：

1. 输出上限不是越大越好
2. 长会纪要不能指望一次性吃满上下文

### 2. rerun 结果和数据库版本的并存复杂度

rerun 系列不是数据库版本，而是 session 目录里的旁路产物。  
这会带来两个复杂点：

1. App 需要识别并优先展示 rerun
2. 版本下拉要把数据库版本和 rerun 结果一起讲清楚

当前 v0.32 已经把这件事做到了“可用”，但从架构上看，它依然是一个过渡方案。

### 3. Markdown 展示不能偷懒

一开始尝试过：

1. `AttributedString(markdown:)`
2. `WebView`
3. `NSTextView`

结果都在桌面 App 的特定布局里暴露了问题：

1. 结构不完整
2. 空白高度异常
3. 只显示部分内容

最终回到：

1. 先做稳定的块级渲染
2. 再逐步补表格和加粗等语法

这说明在桌面产品里，“Markdown 显示”看似小事，实际上很容易踩 UI 细节坑。

### 4. 录音暂停的显示时长和真实时长不一致

这次也暴露了一个很典型的产品细节坑：

1. 底层录音是暂停了
2. 但 UI 计时还在按 `startedAt` 一路增长

所以用户会看到：

1. 暂停时数字还在跑
2. 结束后又被校正

这个问题后来通过：

1. `recordingStartedAt`
2. `recordingPausedAt`
3. `accumulatedPausedDuration`

三者组合才修正。

这类问题的经验是：  
显示层的时间不能偷懒直接用“当前时间减开始时间”，必须和暂停态同步。

## 当前版本边界

v0.32 虽然已经明显强于早期 MVP，但仍有明确边界：

1. 会议纪要 Markdown 渲染目前是产品内定制渲染器，不是完整 Markdown 引擎
2. 长会议纪要仍未做“分段总结 -> 二次汇总”正式闭环
3. diarization 和 ASR 的质量仍受录音输入条件影响
4. rerun 结果仍是文件侧产物，不是严格纳入数据库 schema 的正式版本
5. 环境仍未做内聚打包，依赖本机 Python / ffmpeg / 模型缓存

这些都属于后续版本的明确工作，而不是当前版本的临时疏漏。

## 对下一阶段的建议

### 1. 继续做 V3.2 收口

最值得继续打磨的是体验层的一致性：

1. 版本切换的语义再收紧
2. 纪要渲染语法再补齐
3. 编辑后版本创建的反馈更明确
4. 队列状态与处理阶段展示更可视化

### 2. 推进 V3.3 环境内聚

当前产品已经到了值得做“可交付形态”的阶段。

下一阶段应该考虑：

1. 内置 Python runtime
2. 内置 ffmpeg / ffprobe
3. 固化依赖版本
4. 模型检查与首次下载引导

这样产品才真正具备“给别人安装就能用”的条件。

### 3. 长会纪要要进入双阶段总结

如果后续继续支持更长会议，Gemma 这条链建议升级为：

1. 分段总结
2. 二次汇总

而不是继续把完整长 transcript 一次性喂给模型。

## 当前结论

MVP v0.32 的价值，不是“又多了几个按钮”，而是把产品从一套本地 AI 工作流，推进成了一个开始具备系统能力、历史能力、版本能力和可恢复能力的桌面应用。

本版本最重要的成果有四个：

1. 多会议任务队列成立了
2. 历史记录和版本体系成立了
3. 长会议主链开始可用
4. 逐字稿编辑、回放、浮层这些高频体验能力成立了

如果说 v0.2 解决的是“模型链能不能跑”，  
那么 v0.32 解决的是“这个产品能不能持续被用”。
