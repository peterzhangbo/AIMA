# MeetingSummaryApp MVP 实践总结

## 目标达成情况

本次 MVP 的核心目标已经完成：

1. 本地 macOS 原生应用可启动
2. 可请求麦克风权限与系统音频权限
3. 可同时采集麦克风音频与系统音频
4. 可将双轨音频混音后交给 Whisper 生成 transcript
5. 可使用 Gemma 在本地生成会议纪要 Markdown
6. 所有运行产物统一落在 `文稿/MeetingSummaryApp/sessions/`

当前 MVP 已经具备“录制一段会议音频并生成会议纪要”的基本闭环。

## 当前技术方案

当前版本采用以下架构：

1. 前端形态：SwiftUI 原生 macOS App
2. 麦克风采集：`AVAudioRecorder`
3. 系统音频采集：`ScreenCaptureKit` 音频输出
4. 双轨混音：`ffmpeg`
5. 转写模型：`mlx-community/whisper-large-v3-turbo`
6. 纪要模型：`mlx-community/gemma-4-26b-a4b-it-4bit`

当前主链音频策略：

1. 麦克风保留原始轨：`raw_microphone.wav`
2. 系统音频保留正式轨：`raw_system.wav`
3. 生成普通混音：`raw_capture.wav`
4. 生成转写专用混音：`raw_capture_transcribe.wav`
5. Whisper 固定使用 `--language zh`

当前转写主链规格：

1. 最终用于转写的 WAV：`16000 Hz`
2. 单声道：`mono`
3. 位深：`16 bit`

## 这次实践中做对的事情

1. 先把环境核清楚，再进入实现。这个顺序是对的。
2. 很早就把所有运行产物收敛到 `文稿/MeetingSummaryApp`，后续排障成本明显下降。
3. 很早补了 `session.log` 和界面执行日志，否则后面很多问题根本没法定位。
4. 没有把 Whisper 和 Gemma 包成远程服务，而是直接本地子进程/本地模型调用，减少了很多无关复杂度。
5. 把“普通回放音频”和“转写优化音频”分开，是对后续产品质量有帮助的设计。

## 这次实践里踩到的主要坑

### 1. 录音成功不等于状态机判断成功

前期出现过“文件已经生成，但 UI 判失败”的情况。问题不在录音本身，而在停止录音后的状态判断和文件落盘时机。

结论：

1. 对媒体文件不能只看一次存在性检查
2. 要等待写入完成
3. 要通过日志把停止录音、回调、落盘几个阶段分开记录

### 2. App 内运行环境和终端环境不一样

前期 Whisper 在终端可用，但 App 内调用失败。根因是 App 进程里找不到 `ffmpeg`。

结论：

1. 不能假设 GUI App 继承用户 shell 的 `PATH`
2. 模型链路所需依赖必须显式注入路径

### 3. Whisper JSON 并不总是标准 JSON

最近一次明显暴露了这个问题。Whisper 输出 JSON 中可能包含 `NaN`，Python 可以读取，但 Swift 的 `JSONSerialization` 会直接报格式错误。

结论：

1. 不能把模型输出默认当成严格标准 JSON
2. 解析前必须做清洗或宽容解析

### 4. Gemma 推理入口不能只看“能不能跑”

曾经把 prompt 改成文件输入后，顺手改成了直调 Python API，结果输出乱码。根因不是 prompt 文件，而是漏掉了 CLI 内部使用的 chat template 处理。

结论：

1. “能跑通”不等于“调用方式正确”
2. 本地模型调用要尽量贴近官方 CLI 的真实路径

### 5. 系统音频实时写入链路不能随便降规格

把最终转写输出收敛到 `16k/mono/16bit` 是合理的，但把系统音频实时写入链路也直接降成低规格，会导致 `AVAssetWriter` 报“无法编码媒体”。

结论：

1. 临时采集链路优先稳
2. 最终输出链路再做规格收敛

## 当前 MVP 的已知边界

这版 MVP 可用，但还不是最终可发布产品。当前边界包括：

1. 仍然依赖本机已有模型环境
2. 系统音频与麦克风同时录制时，仍可能受扬声器回灌影响
3. 中英混说场景下，固定 `zh` 提示比自动识别更稳，但不是完美方案
4. 尚未做说话人分离
5. 尚未做历史列表、设置页、失败重试策略、结果比对等产品能力

## 对下一阶段的建议

下一阶段最值得做的不是扩功能，而是提高稳定性与可解释性：

1. 继续收敛录音质量，尤其是回响与双轨对齐
2. 引入说话人分离能力，为后续多人识别铺路
3. 增加结果质量检测，比如重复句、异常语言、异常短输出告警
4. 增加历史记录页，便于回看所有 session
5. 增加设置项，让语言、音频权重、输出格式可以有限可控

## 当前 MVP 结论

这一版已经证明了产品方向成立：

1. 本地双音频录制可行
2. 本地 Whisper 转写可行
3. 本地 Gemma 纪要生成可行
4. 原生 App 形态适合这个产品

从工程视角看，这一版已经是一个真实的 MVP，而不是只会演示的半成品。
