# MeetingSummaryApp 设计输入版 PRD

## 1. 文档目的

本文档基于 `docs/02` 到 `docs/11` 的现有内容，整理出一份可直接用于 Figma 出图的产品需求文档。

目标不是复述所有技术实现细节，而是统一以下设计输入：

1. 当前产品基线是什么
2. 本轮设计稿需要覆盖哪些页面与状态
3. 哪些能力已经成立，哪些仍属于后续增强
4. 设计时必须体现的产品语义是什么

当前产品基线以 `MVP v0.33` 为准，兼容 `V3.1 + V3.2` 已成立能力。

---

## 2. 产品定义

### 2.1 产品名称

`MeetingSummaryApp`

### 2.2 产品定位

一款运行在 macOS 上的本地会议录音与会议纪要工具。

它不是命令行包装器，而是面向真实会议场景的桌面产品。核心价值是把“录音、转写、多人逐字稿、纪要生成、回看与修订”收敛到一个连续工作流中。

### 2.3 产品形态

1. macOS 原生桌面应用
2. 主窗口长期承载历史与详情
3. 录音过程中支持自动切换到悬浮录音浮层

### 2.4 当前版本目标

当前版本已经从“单次录音 demo”演进为“可持续使用的桌面会议系统”，本轮 Figma 设计应围绕以下三个产品主题展开：

1. 录音可靠
2. 历史可管理
3. 结果可消费、可修订、可追溯

## 3. 用户与使用场景

### 3.1 核心用户

1. 需要频繁记录会议内容的个人知识工作者
2. 需要会后快速形成纪要、待办和结论的团队成员
3. 需要回放、修订、再生成纪要的重度会议用户

### 3.2 核心场景

1. 用户开始一场线上会议录音，同时采集系统声音与麦克风
2. 录音结束后，会议进入后台处理，用户可以继续录下一场
3. 用户在历史列表中检索某场会议
4. 用户在详情页查看多人逐字稿与会议纪要
5. 用户点击逐字稿回放对应音频片段，校对内容
6. 用户就地编辑逐字稿，生成更可信的工作材料
7. 用户基于当前逐字稿重新生成新的纪要版本
8. 用户在异常场景下看到明确状态，而不是被底层失败细节淹没

---

## 4. 产品目标与非目标

### 4.1 本轮设计目标

1. 建立完整的桌面信息架构，而不是单页工具界面
2. 清晰表达“当前录音 / 后台处理 / 历史资产 / 详情消费”的关系
3. 将“版本、回放、编辑、重试、降级”这些真实使用语义设计清楚
4. 结果页从工程面板感，收敛为长期可工作的内容界面

### 4.2 非目标

以下内容不应作为本轮主设计范围：

1. 云同步与账号体系
2. 实时转写
3. 实时纪要生成
4. 多模型切换与复杂设置中心
5. 真实姓名级 speaker 识别
6. 完整环境安装引导流程

---

## 5. 已成立能力边界

### 5.1 已成立能力

1. 系统声音与麦克风双源录音
2. 录音结束后自动进入处理队列
3. 本地 Whisper 转写
4. speaker diarization 与多人逐字稿生成
5. 本地 Gemma 生成会议纪要
6. 历史记录中心
7. transcript / summary 版本化
8. 悬浮录音浮层
9. 逐字稿点击回放
10. 逐字稿内联编辑
11. 失败重试与从恢复点继续
12. 系统音频中断后的部分降级

### 5.2 当前仍有边界的能力

设计中需要通过文案和状态体现“有能力但不绝对稳定”的事实：

1. 系统音频采集仍可能受桌面环境影响
2. 会议窗口识别仍是启发式
3. “仅麦克风继续录制”属于降级保底方案
4. 历史标题提炼已可用，但不是严格结构化命名系统
5. rerun 结果与正式版本体系并存，语义仍需收口

---

## 6. 核心产品原则

1. 简洁优先：不要把工程复杂度直接暴露给用户
2. 连续工作流：开始录音、后台处理、历史查看、结果修订要构成一个闭环
3. 状态可解释：用户必须知道正在发生什么，以及失败卡在哪
4. 内容优先：详情页以逐字稿和纪要内容消费为核心，而不是调试面板
5. 桌面效率：历史检索、快捷操作、复制与回放要符合高频使用预期
6. 部分降级优于整场失败：录音链路出现局部问题时，仍尽量保住结果

## 7. 信息架构

### 7.1 一级结构

主产品建议采用三段结构：

1. 左侧：历史记录区
2. 中间：当前会议详情主内容区
3. 浮层态：独立的轻量录音控制入口

### 7.2 页面/视图清单

本轮 Figma 需要至少覆盖以下视图：

1. 首次启动权限页
2. 主窗口空态
3. 主窗口录音控制台
4. 录音中悬浮浮层
5. 处理中的会议态
6. 历史列表态
7. 会议详情页
8. 逐字稿编辑态
9. 版本切换态
10. 错误/失败态
11. 归档缺失态
12. 空结果态

## 8. 页面级需求

### 8.1 权限页

页面目标：

1. 让首次用户清楚知道缺少哪些权限
2. 引导完成授权，而不是只显示失败
3. 在进入主功能前完成必要环境检查

页面内容：

1. 产品价值说明
2. 权限项列表
3. 每项权限状态
4. 去系统设置按钮
5. 重新检测按钮
6. 环境准备状态
7. 进入应用按钮

关键要求：

1. 权限状态要分清 `未知 / 已授权 / 未授权`
2. 禁止只给技术报错，不给行动建议

### 8.2 主窗口录音控制台

页面目标：

1. 成为用户开始新会议的主入口
2. 同时让用户看到后台处理负载

页面内容：

1. 当前录音状态
2. 开始录音 / 停止录音
3. 暂停 / 继续
4. 音源状态
5. 录音时长
6. 后台处理中会议数量
7. 最近会议或历史入口

关键要求：

1. 要清晰区分整体录音状态与分音源状态
2. 当系统音频中断但麦克风仍正常时，主态仍应是“录音中”

### 8.3 悬浮录音浮层

页面目标：

1. 在录音过程中提供最小但高频的控制入口
2. 不阻断用户继续参与会议

页面内容：

1. 录音时长
2. 当前录音状态
3. 暂停
4. 继续
5. 停止
6. 展开主窗口

关键要求：

1. 浮层与主窗口语义一致
2. 重点突出当前是否正在录音、是否发生降级

### 8.4 历史列表

页面目标：

1. 支撑长期使用下的会议检索
2. 让用户快速定位目标会议

页面内容：

1. 搜索
2. 状态筛选
3. 时间快捷筛选
4. 精确日期筛选
5. 历史条目列表
6. 开始新会议入口

单条历史需包含：

1. 会议标题
2. 录制时间
3. 状态标签
4. 简要副信息

关键要求：

1. 时间展示采用录制时间，不采用最后更新时间
2. 标题优先显示内容提炼主题，不长期停留在默认时间标题
3. 搜索、状态、时间筛选支持组合使用

### 8.5 处理态

页面目标：

1. 让用户明确知道会议正在后台哪个阶段
2. 降低“卡住了还是在跑”的不确定感

需覆盖阶段：

1. 排队中
2. 保存音频
3. 转写中
4. 说话人处理
5. 清洗整理中
6. 生成纪要中
7. 已完成
8. 失败

关键要求：

1. 处理态应支持出现在历史项、详情页和录音控制区
2. 不暴露底层命令行噪音

### 8.6 会议详情页

页面目标：

1. 成为用户消费结果、校对内容、继续加工的主工作区
2. 同时承载“版本、回放、编辑、复制、再生成”

页面结构建议：

1. 顶部工具区
2. 左侧多人逐字稿区
3. 右侧会议纪要区

顶部工具区建议包含：

1. 会议标题
2. 会议状态
3. transcript 版本切换
4. summary 版本切换
5. 新建纪要
6. 重试处理
7. 打开文件目录

逐字稿区能力：

1. 段落列表展示
2. 说话人标签
3. 时间范围
4. 点击回放
5. 当前播放高亮
6. 就地编辑
7. 合并到上一段
8. 拆分段落

纪要区能力：

1. 结构化 Markdown 内容展示
2. 复制纪要
3. 空态说明
4. 错误说明

关键要求：

1. transcript 和 summary 都是“版本化内容”，不能被设计成一次性静态结果
2. “重试处理”与“新建纪要”必须分清语义
3. 复制按钮、空态、边距和标题区要统一

## 9. 核心对象定义

### 9.1 会议

一场独立录音与处理任务，是历史列表中的最小单位。

### 9.2 多人逐字稿

带 speaker 和时间信息的结构化记录，是纪要生成的数据基础，也是用户校对和编辑的工作材料。

### 9.3 会议纪要

基于当前逐字稿版本生成的结构化总结结果。

### 9.4 版本

同一会议下，不同阶段或不同操作产生的 transcript / summary 内容快照。

## 13. 待确认问题

这些问题不阻塞先出 Figma 草稿，但在高保真前最好定清：

1. 主窗口默认是“录音控制优先”还是“历史列表优先”
2. transcript 与 summary 的版本切换是分开显示，还是合并到统一版本栏
3. 悬浮浮层是否展示音源级状态细节，还是只展示主状态
4. 失败会议的 CTA 是“重试处理”优先，还是“查看失败详情”优先
5. 归档缺失态是否允许直接从历史中删除记录

---

## 14. 结论

当前 Figma 设计不应再围绕早期 MVP 的“三页演示流”展开，而应围绕已经成立的桌面产品骨架展开：

1. 一个可持续使用的录音入口
2. 一个可检索的历史中心
3. 一个可回放、可编辑、可版本化的详情工作区

这三者构成当前版本最真实、也最值得被设计强化的产品主线。
