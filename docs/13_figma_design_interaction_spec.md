# MeetingSummaryApp 交互文档（Figma 设计输入版）

## 1. 文档目的

本文档用于补足 PRD 中的交互层定义，重点说明：

1. 页面之间如何流转
2. 每个核心页面有哪些状态
3. 用户操作后系统如何反馈
4. 哪些交互语义必须在设计稿中明确体现

本稿以 `MVP v0.33` 为当前交互基线。

---

## 2. 全局交互原则

1. 一级动作显性：开始录音、停止录音、重试处理、新建纪要必须一眼可见
2. 状态优先于装饰：先让用户看懂录音、处理、失败、降级，再谈视觉包装
3. 内容工作区优先：详情页是工作台，不是展示页
4. 反馈即时：复制、编辑、切换版本、重试都要有明确反馈
5. 错误可恢复：失败提示必须尽量附带下一步动作
6. 桌面效率优先：默认支持列表检索、多区域并行查看、减少弹窗打断

---

## 3. 全局状态体系

### 3.1 录音状态

主状态：

1. 未开始
2. 启动中
3. 录音中
4. 已暂停
5. 停止中
6. 录音失败

分音源状态：

1. 麦克风正常
2. 系统音频正常
3. 系统音频中断
4. 当前仅继续录制麦克风

交互要求：

1. 主状态与分音源状态必须分层展示
2. 不能因为系统音频中断，就把主状态直接显示为整场失败

### 3.2 处理状态

1. 排队中
2. 保存音频
3. 转写中
4. 说话人处理中
5. 逐字稿整理中
6. 纪要生成中
7. 已完成
8. 失败

交互要求：

1. 历史项状态、详情顶部状态、队列总量提示应语义一致
2. 失败后允许继续进入详情查看已有内容或发起重试

### 3.3 内容状态

对 transcript 与 summary 两块内容，统一采用以下语义：

1. 无内容
2. 处理中
3. 有内容
4. 内容为空
5. 加载失败
6. 归档缺失

## 4. 全局页面流

### 4.1 首次使用流

1. 用户启动应用
2. 进入权限页
3. 检查权限与环境
4. 权限满足后进入主窗口

### 4.2 新会议主流程

1. 用户在主窗口点击开始录音
2. 进入录音中状态
3. 主窗口可自动收起为浮层
4. 用户暂停 / 继续 / 停止
5. 录音结束后会议进入处理队列
6. 历史列表出现新会议
7. 用户打开详情页查看处理进度或结果

### 4.3 已有会议查看流

1. 用户在历史列表搜索或筛选
2. 选择某场会议
3. 在详情页查看当前 transcript 与 summary
4. 根据需要复制、回放、编辑或重新生成

### 4.4 异常恢复流

1. 会议处理失败
2. 历史项与详情页展示失败态
3. 用户点击重试处理
4. 旧错误提示应清理
5. 会议重新回到排队或处理中状态

## 5. 页面交互说明

### 5.1 权限页

#### 页面元素

1. 顶部产品说明
2. 权限列表
3. 每项权限状态
4. 前往系统设置
5. 重新检测
6. 环境检查状态
7. 继续进入应用

#### 交互规则

1. 点击“前往系统设置”后，不强制跳页关闭当前页面
2. 点击“重新检测”后刷新权限状态
3. 所有必需项满足前，“继续进入应用”不可用
4. 某项失败时，要显示行动型文案，而非单纯红字

#### 需出状态

1. 初始检测中
2. 部分未授权
3. 全部通过
4. 环境检查失败

---

### 5.2 主窗口录音控制台

#### 页面元素

1. 当前录音状态区
2. 主录音按钮
3. 暂停 / 继续 / 停止
4. 音源状态
5. 录音时长
6. 后台处理中会议数量

#### 交互规则

1. 未开始时，主按钮文案为“开始录音”
2. 录音中时，主按钮切换为停止入口，并出现暂停能力
3. 暂停后，时长停止增长
4. 恢复后，从暂停前累计时长继续计算
5. 当系统音频中断但麦克风仍录制时，界面提示“当前仅继续录制麦克风”
6. 停止录音后，不应立即把用户甩到空白页，而应能看到会议进入处理态

#### 需出状态

1. 空闲可录制
2. 启动中
3. 录音中
4. 暂停中/已暂停
5. 停止中
6. 部分降级录音中
7. 启动失败

---

### 5.3 悬浮录音浮层

#### 页面元素

1. 紧凑状态文本
2. 时长
3. 暂停 / 继续
4. 停止
5. 展开

#### 交互规则

1. 开始录音后可自动出现
2. 点击展开，回到主窗口详情
3. 浮层状态必须与主窗口一致
4. 若发生系统音频中断，可用轻提示显示降级状态

#### 需出状态

1. 正常录音
2. 已暂停
3. 仅麦克风继续

---

### 5.4 历史列表

#### 页面元素

1. 搜索框
2. 状态筛选
3. 时间快捷筛选
4. 精确日期筛选
5. 历史列表
6. 开始新会议按钮

#### 列表项信息

1. 标题
2. 录制时间
3. 状态标签
4. 简短摘要或辅助信息
5. 当前选中态

#### 交互规则

1. 搜索输入实时过滤列表
2. 状态筛选与时间筛选可叠加
3. 点击列表项，右侧切换到对应详情
4. 当前选中项必须有明显高亮
5. 处理中的会议状态要能在列表中被快速识别

#### 需出状态

1. 无会议空态
2. 正常列表态
3. 搜索无结果
4. 筛选无结果
5. 失败会议存在

---

### 5.5 处理态

#### 展示位置

1. 历史列表项
2. 详情页顶部
3. 录音控制台中的处理中数量

#### 交互规则

1. 同一会议在不同位置的处理状态文案应统一
2. 若当前已有 transcript 但 summary 仍在处理中，要允许先查看已产出内容
3. 若处理失败，需保留失败原因摘要与重试入口

#### 推荐表现

1. 列表中用状态标签表达进度
2. 详情页顶部用阶段条或多阶段状态卡表达更细进度

---

### 5.6 会议详情页

#### 页面结构

1. 顶部标题与工具栏
2. 左侧多人逐字稿区
3. 右侧会议纪要区

#### 顶部工具栏元素

1. 会议标题
2. 当前状态
3. transcript 版本下拉
4. summary 版本下拉
5. 新建纪要
6. 重试处理
7. 打开文件目录

#### 顶部工具栏规则

1. 已完成会议优先展示“新建纪要”
2. 失败会议优先展示“重试处理”
3. 当会议正在处理中时，重试入口不可与处理中语义冲突
4. 版本切换后，内容区域整体刷新

---

### 5.7 多人逐字稿区

#### 页面元素

1. 段落列表
2. speaker 标签
3. 时间范围
4. 内容文本
5. 编辑入口

#### 交互规则

1. 点击某段，音频跳转到对应时间
2. 播放过程中，当前对应段高亮
3. 点击编辑后就地进入编辑态
4. 编辑失焦自动提交，或显式确认提交
5. 支持修改 speaker
6. 支持修改文本
7. 支持合并到上一段
8. 支持拆分段落

#### 需出状态

1. 只读浏览态
2. 当前播放高亮态
3. 编辑中态
4. 无逐字稿空态
5. 逐字稿生成失败态

---

### 5.8 会议纪要区

#### 页面元素

1. 区域标题
2. 复制按钮
3. Markdown 结构化内容
4. 空态说明
5. 错误提示

#### 交互规则

1. 复制按钮固定在区域右上角
2. 点击复制后给出轻量成功反馈
3. 切换 summary 版本后，内容同步切换
4. 当纪要为空时，优先提示“可新建纪要”
5. 当纪要生成失败时，优先提示可重试或重新生成

#### 需出状态

1. 有内容
2. 处理中
3. 空态
4. 错误态

---

## 6. 版本交互规范

### 6.1 transcript 版本

当前语义：

1. 默认展示最新版本
2. clean 基线版本可显示为 `v1`
3. edited 版本顺延编号
4. rerun 结果可能与正式版本并存

交互要求：

1. 版本切换用下拉而非平铺
2. 当前版本要有明确选中标识
3. 不应把 raw 级中间态直接当主用户版本暴露

### 6.2 summary 版本

当前语义：

1. 同一会议可基于当前 transcript 生成多个纪要版本
2. 新建纪要产生新 summary version

交互要求：

1. “新建纪要”应被理解为“基于当前逐字稿再生成一个新版本”
2. 不能与“重试处理”混淆

---

## 7. 异常与边界交互

### 7.1 系统音频中断

表现要求：

1. 主状态保留“录音中”
2. 次级提示显示“系统音频已中断”
3. 给出“当前仅继续录制麦克风”的明确说明

### 7.2 处理失败

表现要求：

1. 历史项显示失败状态
2. 详情页显示失败信息摘要
3. 提供重试处理入口
4. 重试开始后清理旧错误展示

### 7.3 归档文件缺失

表现要求：

1. 明确告知“历史索引存在，但归档文件缺失”
2. 提供清除此记录入口
3. 不伪装成空态或加载失败

### 7.4 无内容空态

表现要求：

1. transcript 空态与 summary 空态风格统一
2. 空态要说明“为什么没有内容”与“接下来能做什么”

---

## 8. 反馈与微交互建议

1. 复制成功：轻提示，不打断主流程
2. 切换版本：内容区有短暂加载反馈
3. 开始录音：主按钮状态立即切换
4. 停止录音：出现明确的“已进入处理”反馈
5. 编辑提交：局部成功反馈，避免全页刷新
6. 重试处理：旧错误态清空，并回到排队/处理中状态

## 10. 结论

这份交互文档的重点，不是把所有边角逻辑都写满，而是把当前产品最关键的几个语义钉牢：

1. 录音不再是一次性动作，而是一个可降级的连续过程
2. 历史不再是附属页，而是长期使用的资产入口
3. 详情不再是结果展示页，而是回放、编辑、版本化加工的工作台

如果 Figma 设计能把这三点表达清楚，设计稿就已经贴近当前产品真实阶段，而不是停留在早期 MVP 演示层。
